Utilice estas buenas prácticas para prevenir problemas en los flujos de trabajo de Git y para facilitar la resolución de problemas cuando se produzcan.

Errores comunes de Git y cómo solucionarlos
'fatal: negarse a fusionar historiales no relacionados'
Este error se produce cuando un desarrollador intenta combinar dos proyectos no relacionados en una sola rama. Este problema surge cuando la rama contiene historiales de confirmación inconsistentes y etiquetas con la solicitud de extracción y la clonación.

Existen dos maneras de corregir los fatal: refusing to merge unrelated historieserrores.

Primer acercamiento:

El primer método para solucionar este problema es usar la siguiente bandera de git: –allow-unrelated-histories.

Para lograr esto, puedes usar el comando git: git pull origin master --allow-unrelated-histories .

Usando un repositorio remoto, puedes reemplazar origincon su nombre. También puedes sustituir la masterrama con cualquier otra rama que desees integrar en la solicitud de extracción. El propósito de —-allow-unrelated-historieses permitir la fusión de ramas no relacionadas. Esta bandera de Git funciona sin problemas en ausencia de conflictos de archivos. Sin embargo, surge al menos un problema, y ​​debes usar el método de resolución estándar de Git para solucionarlo.

Segundo enfoque:

El segundo método para solucionar este problema consiste en seguir los siguientes pasos: 

Primero, elimina los commits existentes.
Luego, guárdalos.
Clona el repositorio remoto correspondiente y, a continuación, coloca el contenido de tu rama guardada en el nuevo clon.
Esto garantizará que cualquier posible conflicto de código se resuelva antes de la fusión y evitará que se produzcan errores en la aplicación. 

El comando git: git reset HEAD~eliminará todos los archivos de staging de tu último commit. Usando el comando git git stash, puedes almacenar tus archivos no guardados.

Esto proporcionará un árbol de trabajo limpio en el que podrá importar su repositorio remoto. Después de importar correctamente a su rama, puede recuperar los archivos, confirmarlos individualmente como una confirmación separada y corregir los conflictos de archivos. Use el comando `git pop` para recuperar los archivos. Esto volverá a aplicar los cambios que ha guardado en la copia de trabajo actual. Obtenga más información en nuestra guía detallada sobre el error fatal al rechazar la fusión de historiales no relacionados .

'fatal: el origen remoto ya existe'
Esto ocurre cuando intentas cambiar la originURL del repositorio después de clonarlo desde GitHub u otro repositorio remoto a tu estación de trabajo local.

Al usar Kubernetes, este problema puede surgir al intentar incorporar repositorios Git a una orquestación. En la mayoría de los entornos de desarrollo, el controlador predeterminado es el origen. A continuación, se presentan tres soluciones para este problema.

Primer acercamiento:

Para solucionar este problema, hay que eliminar remoteel repositorio remoto existente. El repositorio remoto origin indica la ubicación del repositorio remoto. En la mayoría de los casos, el repositorio remoto origines el único puntero a un repositorio local. Si los usuarios desean modificar la URL de referencia asociada con el repositorio remoto origin, pueden eliminar el repositorio remoto existente originy volver a agregarlo con la nueva URL. Use el removecomando en el repositorio remoto remotey luego ingrese el nombre del controlador (en este caso, origin) para eliminar el controlador.

Primer acercamiento:

Podemos solucionar este problema actualizando la URL existente remote. No siempre es necesario eliminar el controlador de origen del servidor remoto. Una solución eficaz para corregir este error es modificar la URL de referencia del controlador. Para ello, ejecute el set-urlcomando, seguido del nombre del controlador (en nuestro caso, origin) y, por último, la nueva URL.

Segundo enfoque:

Podemos solucionar este problema cambiando el nombre del remoto existente. Alternativamente, se podría cambiar el originnombre a otro diferente. Esto indica que, en lugar de eliminar la URL de apuntamiento del controlador para crear un espacio para la nueva, se puede cambiar el nombre conservando los datos originales. Para lograr esto, ejecute el rename comando con remote. Obtenga más información en nuestra guía detallada sobre el error fatal de origen remoto ya existe .

'Problema con el certificado SSL: no se puede obtener el certificado del emisor local'
Es un error común de Git que encuentran los usuarios al intentar enviar, recibir o clonar un repositorio Git con Git Bash, una herramienta de línea de comandos específica de Windows. Alternativamente, se podría cambiar el nombre origina un nombre diferente. Indica que en lugar de eliminar la URL que apunta al controlador para crear un lugar para el nuevo, se puede cambiar el nombre conservando los datos originales. Para lograr esto, ejecute el comando de cambio de nombre con el remoto. Cuando el certificado SSL utilizado por el servidor Git es autofirmado, los usuarios encuentran un error que indica: unable to get local issuer certificate. Esto se debe a que la clave privada vinculada a él es una vulnerabilidad de seguridad que no se puede revocar. Cuando un certificado autofirmado crea un error de certificado SSL, agréguelo al almacén de certificados de confianza para solucionarlo.

La ubicación predeterminada de Git Bash para el almacén de certificados de confianza es la siguiente:

C:Program FilesGitmingw64sslcerts

Abre ca-bundle.crtel directorio mencionado anteriormente y añade el certificado SSL de Git. Una vez hecho esto, guarda el archivo y ejecuta los siguientes comandos: `git pull`, `git push` o `git clone`. Por motivos de seguridad, no se recomienda desactivar la validación del certificado SSL. Sin embargo, puede ser una solución a este problema.

El siguiente comando desactiva la validación del certificado SSL en Git localmente:

$ git -c http.sslVerify=false clone [URL]

Para solucionar este error, reinstale Git utilizando la configuración de backend de transporte SSL. Obtenga más información en nuestra guía detallada sobre el problema del certificado SSL: no se puede obtener el certificado del emisor local .

'No logró presionar a algunos árbitros para que...'
Esto ocurre cuando un desarrollador sube código a un repositorio Git externo. El proceso de subida de código deja de funcionar repentinamente. Este error se produce cuando no se confirman las modificaciones antes de subirlas, hay problemas con el hook de pre-subida de Git o el nombre de la rama es incorrecto. También se produce cuando varias personas actualizan la misma rama simultáneamente y el repositorio remoto tiene cambios más recientes que tu máquina local.

Estas son las causas y soluciones del failed to push some refserror:

Otro desarrollador envió una confirmación a la misma rama.

El error de terminal aparece de la siguiente manera:

A git@git.testdomain.com :sometest.git ! [rechazado] your-branch -] your-branch (no-fast-forward)
Git no puede gestionar esto porque el nodo head se encuentra en la misma línea de tiempo del código en varios puntos. Como resultado, el repositorio origin es más reciente que tu copia de trabajo actual. Para solucionarlo, actualiza tu repositorio local con git pull. Esto debería restaurar el push de origin.

git pull origin [tu-rama] git push origin [tu-rama]
Obtuviste un error 'master (non-fast-forward)' con un error 'failed to push some refs to'.

Se produce un error fast-forward cuando se avanza el puntero de referencia del historial de confirmaciones. Si tu código se desvía antes de la última confirmación, no podrás enviar ciertas referencias y se producirá un error. --rebaseEl indicador puede solucionar este problema. --rebase Mueve los archivos de confirmación al código de extracción más reciente.

A continuación se describe cómo extraer utilizando --rebase:

git pull --rebase origin [branch]

Obtuviste un error 'master (fetch first)' con un error 'failed to push some refs to'.

Cuando sucede, alguien ya ha enviado cambios a la rama. Git requiere un pull antes de enviar tu modificación. stashRealiza modificaciones locales para evitar perder trabajo durante el proceso pull. Para enviar modificaciones locales, usa —-forcepara solucionar este problema. Sin embargo, usar —-forcepuede producir inconsistencias. Usa —-rebasepara actualizar tu repositorio local sin crear una divergencia en el repositorio remoto.

Usar —-force para solucionar este error puede causar más problemas. Esto se debe a que —-force emplea brute force para hacer que tu código actual y refhead sean la fuente de verdad. Como resultado, tu push puede sobrescribir los cambios del repositorio remoto, eliminando cualquier característica o mejora que otros desarrolladores hayan agregado. Úsalo —-forcesi no te importa que se sobrescriban las características locales. Úsalo —-force si tu repositorio local es correcto. 'fatal: no es un repositorio git'
Todos los comandos de Git están diseñados para ejecutarse junto con un repositorio. Git no sabrá qué enviar si lo ejecutas push -u origin mastersin un repositorio. Este error significa que ejecutaste un comando específico del repositorio fuera de Git. A pesar de ser un problema común, es fácil de solucionar. Este problema significa que no estás en un repositorio Git. Hay dos posibles causas:

Intentaste ejecutar el comando sin navegar a la carpeta del proyecto que contiene el repositorio git.
Te encuentras en la carpeta del proyecto, pero no has inicializado el repositorio Git para esa ubicación.
Para solucionar el primer problema, revise la carpeta desde la que intenta ejecutar el comando. Luego, utilice el cd comando para acceder al directorio correcto si necesita estar en él. En el segundo problema, el usuario debe inicializar el repositorio Git en la carpeta del proyecto. Para ello, navegue hasta la carpeta correspondiente y ejecute el comando git initpara crear un repositorio Git nuevo o reinicializarlo.

'fatal: repositorio no encontrado'
Repository Not FoundEste error se produce al clonar o enviar modificaciones a un repositorio. Puede indicar que el repositorio no existe o que se eliminó. El repositorio existe, pero un problema de acceso o de origen remoto provoca este error.

No te autenticaste

Si intentas acceder private GitHub or BitBucket repositorysin autenticarte, recibirás un error de repositorio no encontrado. Por lo tanto, inicia sesión en el repositorio con tus credenciales en la URL de Git para autenticarte:

git clone https://mcnz: githubpass@github.com /cameronmcnz/private-github-repo.git
No eres un colaborador

Aunque puedas autenticarte en GitHub o GitLab, seguirás recibiendo este error si no eres colaborador del proyecto. Ponte en contacto con un GitHub or BitBucket repositoryadministrador y solicita que te añadan como colaborador.

Mayúsculas incorrectas o una palabra mal escrita

El nombre del repositorio puede distinguir entre mayúsculas y minúsculas si la herramienta de gestión de código fuente se ejecuta en Linux. Presta atención a las grafías poco convencionales del repositorio, como 0 en lugar de O o 1 en lugar de L. Si es posible, copia y pega el código de `git clone` de la documentación.
